Title: GDB

Debug 的基本原则是确认自己假设为真的条件为真，更确切的讲，可以归纳出以下几条原则：

  • Start Small；

  • Use a top-down approach；

  • Use a debugging tool to determine the location of a segmentation fault；

  • Determine the location of an inﬁnite loop by issuing an interrupt；

  • Use binary search；

用 GDB 进行调试的几个基本方式也可归结如下：

  • 在源码中暂停及步进：

    • 断点：braekpoint , watchpoint , catchpoint；

    • 单步：next , step；

    • 恢复执行：comtinue , until , finish；

    • 临时断点：tbreak；

  • 查看变量：print；

  • 全境通告：watch；

  • 函数调用栈；

“常识”

这部分放置的是 GDB 中的常识

  • GDB 遇到断点时打印的代码是将要执行的代码，而不是已经执行的代码；

  • GDB 正常工作的基础是，生成的可执行文件中包含有 debug 信息（symbol table），在 gcc 中则需要加上 -g 选项；

  • 尽管有一种断点就称为 breakpoint，但是断点的统称也被称为 breakpoint；

  • 最好不要对已经被编译器优化过的代码使用 GDB，这可能会导致 GDB 的行为发生偏差（尤其是在断点指定的精确性上）；

  • GDB 中的表达式包含：

    • GDB 中的 convenience variables；

    • 程序中任何 in-scope 变量；

    • 任何类型的常数、字符串、字符常量；

    • 程序预处理宏；

    • 使用源文件语言定义的条件语句、函数调用、强制类型转换、操作符；

  • 后面这一节中提到的“变量”，指代 GDB 当前所选栈帧对应函数中能访问到的变量；

考虑到每次 GDB 退出时，都会清除所有设置，因此需要指定 GDB 的 Startup file：

  • .gdbinit：位于家目录中的.gdbinit文件在程序被加载之前读取，位于当前目录中的.gdbinit在程序被加载之后读取，因此应当将仅和当前程序相关的指令放在当前目录中的文件，通用的配置命令设置在家目录的文件中；

  • gdb -command=<file_a> <file_b>：在 GDB 执行 file_b 之前先执行 file_a；

断点

断点的编号从 #1 开始，有三种常用的断点断点-1：

  • breakpoint：最常见的断点，当 GDB 执行到相应位置时就会停下；

  • watchpoint：当指定内存位置中的值发生改变时，GDB 停下；

  • catchpoint：指定事件发生时，GDB 停下；

在此基础之上，有三种附加性质：

  • tbreak：程序下一次被此断点打断之后，此断点会被删除；

  • hbreak：GDB 会通过硬件的帮助设置此断点；

  • rbreak：接受一个 grep 式的正则表达式，在名字与之匹配的函数中设置断点；

断点的设置位置（location）也非常多变，位置包括以下几种：

  • 当前文件下源代码的行号（可使用 filename: 指定文件）；

  • 行号偏移量（± offset）；

  • 代码的地址；

  • 函数名，默认情况下所有同名函数都涵盖在其中（可使用 filename: 指定文件）；

  • 标签，仅会在当前所选择的栈帧中搜索给定标签（可使用 function: 指定函数）；

此外，考虑到同名问题，还有一种显式位置（explicit location）可供选择：

  • -source filename；

  • -function function；

  • -qualified：如果仅给定 B::fun 作为参数的话，A::B::fun 和 B::fun 都会被选中。此时使用 -qualified 就能仅选中 B::fun；

  • -label label；

  • -line number；

设置断点

一方面，可以通过 break [filename:]funcion 在函数入口处设置断点，不添加filename会在所有文件中的对应函数中设置断点。如果同一个文件中含有多个重名函数，那么可以通过添加参数列表（写出函数的签名）的形式进一步精确指代函数：break function(type)

也可以通过 break [filename:]line_number 的方式在指定行号中设置断点（如果没有添加filename，那么会在焦点文件断点-2中设置断点）

禁用 & 删除断点

有两种删除断点的方式：

  • delete <bp_id>：删除断点标识符为指定值的断点；

  • clear [filename:][function|line_number]：语法和创建断点时使用的语法相同；

有三个和断点禁用相关的指令，它们都接受 breakpoint-list：

  • enable：启用对应断点；

  • disable：禁用对应断点；

  • enable once：类似于tbreak，程序下一次被指定断点阻隔时，将会禁用此断点（但不是删除）；

继续执行

有四种控制断点继续执行的指令：

  • 单步推进，这两者唯一的区别仅在于遇见函数时是否进入函数体。这两者也可以接受一个常数参数，表示直接跳转 n 步：

    • step：遇见函数时，跳转到函数中继续步进；

    • next：遇见函数时，不会跳转到函数中继续步进，而是将函数调用当成一条指令；

  • continue：以断点为单位的跳转，如果未指定参数的话会直接跳转到下一个断点的位置；如果指定常数参数的话，会忽略接下来遇到的 n 个断点；

  • finish：跳转直到当前函数调用栈帧被弹出（不过，如果执行的中途会经过断点，GDB 将会被阻塞）；

  • until location：默认情况是跳转直到当前指令对应机器码行号比当前行号高为止，但是还有着各种各样特别的性质：

    • 跨越循环继续执行-1；

    • 当前栈帧被弹出会停止执行；

    • 跨越多层递归实例，因此 until 仅能设置位于本层栈帧（递归实例）中的位置；

  • advance location：类似于 until，但是不会被限制在本层栈帧中，因此不能用来跳过多层递归实例，两者的不同具体看以下代码：

     94 int factorial (int value)
     95 {
     96 if (value > 1) {
     97 value *= factorial (value - 1);
     98 }
     99 return (value);
     100 }

    如果在 96 行执行 until 99 的话，97 行中的所有派生出的递归实例都会被跳过，直到返回本层递归实例中的第 97 行；反之如果执行 advance 99 的话，在最上层递归实例返回时函数就会被停止，而且沿路的所有递归实例经过第 97 行都会被停止。总的来说，until 的目的地是本层栈帧中的位置，advance 则不会受到栈帧的影响

条件断点

条件断点和 watchpoint 类似，都是用来监视变量而不是程序执行流的。不同于 watchpoint 在指定变量变动的时候就会阻塞 GDB，条件断点只有在程序执行到指定位置且满足指定条件时才会阻塞程序。因此最佳实践是使用 watchpoint 寻找可能出现的代码段，而使用条件断点在该代码段中进一步寻找

条件断点的设置非常灵活，就条件语句来说可包含：

  • 逻辑表达式、算术表达式、Bitwise 表达式；

  • 自定义函数、库函数（皆需连接至程序）条件断点-1；

条件断点的设置方式有两种：

  1. 新建条件断点：break <same args as break> if <condition>；

  2. 给已有断点加上条件：cond <breakpoint id>；

命令列表

考虑到 GDB 被断点阻塞时，接下来的操作通常是执行一些指令，可以使用command将这两项操作合二为一：

 command <breakpoint id>
 > ...
 > commands
 > ...
 end

这一段命令的目的就是在断点标识符为<breakpoint id>的断点阻塞程序时执行commands

 

除了可以将常见的命令放在commands中之外，将以下命令放置在其中有特别的效果：

  • slient：将此命令放置在其中，那么等到 GDB 遇到此断点时将不会打印此断点相关的信息，而仅仅只是执行命令；

  • continue：可以起到跳转指令执行的作用；

当然，如果不想重复的为许多断点输入同一套指令，也可以使用宏解决这个问题：

 define <command name>

定义完成后执行<command name>就拥有和执行定义时输入的命令相同的效果

如果想要查看定义的所有宏，执行show user

如果想要为指定宏编写帮助文档，执行document <command name>

watchpoint

watchpoint 的特殊之处在于，它会全局检测内存位置中值的变化情况，但凡其中的值发生了变化就会阻塞程序。这一特点在很多时候都非常有用，比如说检测被指针指向的指针指向的值，此时由于难以明确其中的值到底是在什么地方被修改的，因此直接使用 watchpoint 全局监视指定内存位置可能是一种更好的选择

需要注意的是，watchpoint 检测的变量必须位于 GDB 当前的作用域之内

检视变量

本小结主要介绍如何使用 GDB 原生的套件以及程序中自带的函数检视程序中各种各样的变量，包括但不限于：

  • 语言基本变量；

  • 自定义结构体或类变量；

  • 内存地址；

  • 变量类型定义信息；

  • 静态与动态数组；

  • 寄存器；

除了下面介绍的 one-shot 类型的指令，此外的还有通过交互方式展示数据的指令 explore <arg>

值与内存

考虑到 GDB 中的很多指令都会将表达式作为参数接收，因此这里有必要详细介绍一下表达式

除了 这里 介绍的类型可以用作表达式，形如 {elem,elem...} 的表达式可以被当成数组类型

GDB 原生也支持一些特别的操作符：

  • <element> @ <length>：这种语法一般用以指代程序中动态分配空间的数组（malloc）值与内存-1，<element>指的是解引用数组地址后得到的值（*array）,<length>指的是数组的长度（元素个数），效果如下所示：

     (gdb) p/x *array@len
     $1 = {0x1234, 0x5678}

  • [file|function]::variable：指代特定作用域中的静态变量；

  • {type} addr：指代地址为 addr 的地方有一个类型为 type 的变量；

Output Format

这部分内容涉及到两个手动打印命令的格式配置问题：

  • p/[fmt] <var>：主要用来打印单个变量，如果希望重复打印变量历史中最后一个变量的话，无需添加<var> 直接执行即可：

    • 将参数当成 hex/dec/unsigned dec/oct/binary/常量字符 的整数打印：x/d/u/o/t/c；

    • 将参数当成地址打印：a；

    • 将参数当成浮点数打印：f；

    • 将参数当成字符串（需为 ’\0‘ 结尾）打印：s；

    • 其他：x,r；

  •  

    • x/nfu addr：主要用来打印内存中连续的多个变量。nfu 是用来指定期望打印的数据块数目以及格式化它们的方式，addr 则是用来指定地址的开始处：

      • n：打印的单元数目（单位为u），如果输入负值，将会从addr倒序打印；

      • f：这部分的参数和[fmt]中的参数相同，不赘述了；

      • u：单元大小，基本上分为四种，每次指定这个参数的时候都会作为下一次执行x时u的默认参数。此外，指定i,s作为[fmt]时，此参数会被省略：

        • b：Byte；

        • h：Halfword；

        • w：Word；

        • g：Double Word；

      • addr：开始打印内存内容的地址；

    当查看机器指令时，a=>表示当前正在执行的指令：

     (gdb) x/5i $pc-6
         0x804837f <main+11>: mov %esp,%ebp
         0x8048381 <main+13>: push %ecx
         0x8048382 <main+14>: sub $0x4,%esp
      => 0x8048385 <main+17>: movl $0x8048460,(%esp)
         0x804838c <main+24>: call 0x80482d4 <puts@plt>

此外，使用 $_ 可以指代 x 最后一次检视的地址，$__可以检视该地址的内容

动态数组

有两种形式的动态数组：

  • 一种是直接在堆中申请一大片区域，直接用来放置期望保存的元素。想要打印这种类型的动态数组只需要使用 这里 提到的语法就可以了；

  • 另一种是将动态数组作为数据的托管对象，此时堆数组存放的不再是元素对象本身恶，而是一些指向它们的指针，这种情况就需要用到 GDB 变量 了；

自动打印

有两种 “程度” 的自动打印：

  • 每次程序执行被阻塞（断点，next 等等）时都会打印一些数据，核心是 automatic display list；

  • 程序被指定断点阻塞时，GDB 执行附着在该断点上的指令，核心是 条件断点；

前者可以理解为在程序每一次停顿的时候都将 auto-list 中处于启用状态的变量、按照设置时的格式、用类似于执行print的方式，打印出来。有三种设置 auto-list 的格式，含义各不相同：

  • display expr：将expr添加到 auto-list 中，停顿时打印；

  • display/fmt expr：[fmt]和这里提供的参数相同，停顿时等价于执行 p/fmt expr；

  • display/fmt addr：[fmt]特指i,s，停顿时等价于执行x/fmt addr；

禁用及删除的指令如下：

  • undisplay <dnums>；

  • disable display <dnums>；

  • enable display <dnums>；

还有一些预览作用的指令：

  • info display；

  • display；

另外，当 auto-list 中出现函数的本地变量时，在脱离该变量的作用域之后会自动禁用此变量，下一次再进入此变量作用域之后又能重新启用

Convenience GDB

这部分内容涉及到很多方面，但是本质上都是围绕着仅在 GDB 中有效的、各种以$开头的函数与变量。通过引入这些组件能解决许多原来不能解决的问题（比如说打印托管容器中的元素），也能方便很多原来较为麻烦的操作

变量历史

GDB 的变量历史（variable history）指的是曾经被print打印过的所有变量的历史（无论是直接还是间接调用print），这些变量会被一直保存到 GDB 重新读取程序的符号表为止；可使用show values打印变量历史

变量历史中出现的变量都会被给予一个以$开头、从$1开始的数字，作为该值的历史编号（history number）：

  • 如果想要引用指定的历史变量，使用$n即可；

  • 如果想要引用最近的历史变量，使用$即可；

  • 如果想要引用 2. 之前的一个变量，使用$$即可（注意，$$0等价于$）；

通过使用p *$.next能将变量历史中最近一次的历史变量作为指向结构体的指针，并将该结构体的下一个结构体对象打印出来；此时，不断地按下回车键就能遍历这整个链表了

需要注意的是，这是变量历史，使用set x=5并不能为这些变量赋值（如果需要赋值的话需要使用 GDB 变量）

GDB 变量

GDB 变量顾名思义是和程序完全无关的变量，仅仅是储存在 GDB 中的变量。这些变量统一以$开头，完整的引用方式为$<name>，需要注意的是，<name>既不能为数字也不能为寄存器保留值

使用如下代码可以为foo这一个 GDB 变量赋值：

 set $foo = *object_ptr

另外，由于 GDB 变量并没有固定的类型，因此也可以这样做：

 set $foo = int_value

除了上述代码，也可以使用此命令为 GDB 变量进行赋值：

 init-if-undefined $variable = expression # 仅当变量不存在时才会赋值，防止覆盖

如果想要查看定义好的 GDB 变量，可以执行：

 show convenience

GDB 变量经常被当做计数器以及指针运算的参与者：

 set $i = 0
 print bar[$i++]->contents # 不断敲击回车就能遍历此数组中的所有内容

另外还有一些已经被设定好的 GDB 变量：$_,$__,$_exitcode

GDB 函数

这些也是一些已经设定好的函数，可以直接被使用在 GDB 表达式中作为运算的参与者，具体参考 GDB 手册

函数调用栈

函数调用栈中依据函数调用和返回的顺序和保存函数本身的信息，其中每一个信息单元和一个函数的调用一一对应，这些信息单元被称为帧（frame），函数调用帧本身的信息包含有：

  • 函数的实际参数：argvs；

  • 函数的局部变量：locals；

  • 函数调用帧在栈中的地址；

GDB 在程序的执行过程中，始终选中一个函数调用帧作为许多命令的默认执行对象（printf <var>），一般被选中的帧是位于函数调用栈最顶上的函数所对应的函数调用帧

下面是一些相关的术语：

  • initial frame & outermost frame：程序入口函数的帧；

  • innermost frame：当前正在执行的函数的帧；

  • frame level & frame number：该帧在调用栈中的位次，通常 innermost frame 的位次是 0，此函数 caller 的 frame number 为 1...以此类推；

  • backtrace：函数调用的轨迹，执行backtrace（bt）即可将程序的整个函数调用栈打印出来；

指定栈帧

常用的有两种指定栈帧的方式，一种是通过frame-selection-spec直接指定目标帧；另一种是通过up或down指定相对于当前选中帧前进或后退多少个帧

无参数执行frame时，会直接打印当前选中帧的信息，这是一种查看当前帧位次的重要方式

直接指定

有四种直接指定帧的方式（即所谓的frame-selection-spec）：

  • <num> & level <num>：函数调用帧的 frame number，由于这种方式很常用的缘故，level可以省略；

  • address <stack-address>：函数调用帧的地址，即(gdb) info frame Stack level 1, frame at 0x7fffffffda30: 中的 0x7fffffffda30；

  • function <function-name>：函数的名字，如果有多个重名函数位于栈中（递归），那么就会选择最内层的函数；

  • view <stack-address>：GDB 不能生成合适的 backtrace（程序拥有多个 backtrace 或者当前 backtrace 被 bug 打断）时，使用这个命令在不同的栈帧中切换就非常合适；

执行frame <frame-selection-spec>就能切换 GDB 当前选中的帧

间接指定

  • up <num>：选取相对于当前指定帧、向着 innermost frame 移动的第 num 个帧；

  • down <num>：选取相对于当前指定帧、向着 outermost frame 移动的第 num 个帧；

静默版本

上述指令（frame，up，down）执行完之后都会输出目标帧的相关信息，相对的也有不输出指定帧相关信息的静默版本：

  • frame select-frame [ frame-selection-spec ]；

  • up-silently <n>；

  • down-silently <n>；

查看栈帧

之前提到直接执行frame能简短的展示的当前焦点栈帧的信息，除此之外使用info f也能起到类似的效果：

  • info frame [ frame-selection-spec ]：显示指定栈帧的详细信息，未指定参数会直接展示当前栈帧的信息；

  • info args [-q] [-t type_regexp] [regexp]：后方的参数是用来筛选当前焦点栈中实际参数的通配符：

    • -q regexp 用来匹配参数的名字；

    • -t type_regexp 用来匹配参数的类型；

  • info locals [-q] [-t type_regexp] [regexp]：作用和 2. 类似，不过是打印焦点栈本地变量的；

批量执行

frame apply [all | count | -count | level level...] [flag]... command 即是用来批量执行命令的语句，此中的批量执行可以理解为将指定函数调用帧当成 #0 帧执行命令，将此函数 Callee 对应帧当成不存在， Caller 对应帧不变

靠前方括号中参数的作用：

  • all：对所有栈帧执行 command；

  • count：对从 #0 开始的 count 个栈帧执行 command；

  • -count：对从 outermost 开始的 count 个栈帧执行 command；

  • level：如2-4 6-8 3指的就是第 2,3,4,6,7,8 这几个帧；

靠后方括号中参数的作用：

  • -c：显示执行命令过程中相关的错误信息，同时继续执行命令；

  • -s：不展示命令产生的错误信息和空输出，也不展示产生错误的栈的相关信息；

  • -q：不展示栈相关的任何信息，仅展示命令的输出信息；

简单打印

就查看机器码而言，简单打印基本上只有以下几种用法

  • print/i $pc ：打印当前正在执行的机器指令，而 x/ni 会打印将要执行的指令；

  • x/ni：考虑到 n 的正负性，这条 gdb 指令实际上会打印从当前正在执行的指令往前/后数n 条机器指令；

  • display/i $pc：这条指令用于将 “当前正在执行的机器指令” 纳入默认打印列表

另外，对于 x 指令来说，考虑到其会将$_作为其默认操作对象，因此连续执行 x/i 的话就能从当前指令开始往下遍历指令

 

disassembly

到这里使用的指令实际上就相对注重机器码和源码之间的映射关系了，考虑到通常情况下一条源码经常对应着多条源代码，因此可以使用 info line location 直接查看这两者之间的映射关系（默认参数是 $pc）：

 (gdb) info line m4_changequote
     Line 895 of "builtin.c" starts at pc 0x634c <m4_changequote> and \
     ends at 0x6350 <m4_changequote+4>.
 (gdb) info line *0x63ff
     Line 926 of "builtin.c" starts at pc 0x63e4 <m4_changequote+152> and \
     ends at 0x6404 <m4_changequote+184>.

值得注意的是，info line 也能将位置作为参数

寄存器

从简单的方面来看，如果将要检视寄存器中的内容的话，无非就是以下三个指令：

  • info register：检视当前寄存器中的值及其姓名，不包括浮点数寄存器和向量寄存器；

  • info all-register：检视所有寄存器；

  • info register register_name：检视指定寄存器中的值；

这些命令都假设指定寄存器中的内容是整数，如果该寄存器中的内容是浮点数，使用 print/f $regname

从复杂的方面来说，考虑到寄存器实际上能被分为两种：saved register 和 temp register，而他们最大的不同在于其中的内容是由谁保证的，如果是前者，寄存器中的内容需要由 Callee 进行保证，也就是说，其中的值在 Callee 对应调用帧中必然能检索到；相对的，如果是后者则其中的值由 Caller 进行保证，其中的值很可能只能在 Caller 中才能检索到

为此，需要根据目标寄存器的类型，谨慎的选择当前的焦点栈帧（frame #num）；特别的，如果想要检索当前寄存器中的实际值，最好先执行 frame 0

步进

如果想要步进机器码的话，执行 nexti 和 stepi

[断点-1]  当这些断点重合时，仅会有一个断点被触发，标识符较小的触发优先级较高
[断点-2]  默认情况下 GDB 的焦点文件是含有main()的文件，但是当以下行为发生时焦点文件将会切换：对其他源文件执行list、步进到其他源文件中、GDB 命中位于其他源文件中的断点
[继续执行-1]  实际上until可以接受和break一样的参数，直接指示跳转的目的地
[条件断点-1]  需要注意的是，如果没有该函数的 Debug 信息（函数所在源文件没有-g），GDB 会默认这个函数的返回值是int类型的数据
[值与内存-1]  除此之外，也能通过强制类型转换的方式打印数组内容：(gdb) p/x (short[])0x12345678
