Title: 网络层：数据平面 : IP 协议

IP（Internet Protocol）具有以下特点：

  • IP 为网络中的数据报提供路由功能：

    • 每个数据报都需要都起始地址；
    • 运行在每个节点上的 IP 协议会判断数据报是否到达其目的地址；
    • 如果并未达到目的地址，IP 需决定该数据报的下一跳；

  • IP 提供的服务是不可靠的（unreliability）：

    • 包不一定会到达指定地址；

  • IP 负责对包进行分段（网络层也能对数据进行分段）：

    • 数据报不能的大小不能超过网络的最大容量（MTU）；
    • 被分割成碎片的数据报必须包含重组所需要的所有信息（长度、偏移量）；

本章将涉及以下几个部分的内容：

  • IPv4 相关：

    • 数据报格式；
    • 编址方式；

  • 网络地址转换（Network Address Transition，NAT）：

    • 域内使用私有地址相互通信；
    • 任何域外通信都需要经由路由器将私有地址转换为外侧地址（体现为域中所有设备共享该外侧地址）；

  • 动态主机配置协议（Dynamic Host Configuration Protocol，DHCP）：

    • 为接入到网络中的设备自动配置 IP 地址；

  • IPv6 相关：

    • IPv6 的帧格式；
    • IPv4 要如何 “兼容” IPv6；

IPv4 编址

因特网中的每个设备至少拥有一个 IP 地址，基于 TCP/IP 协议实现的私有网络也是如此。本质上，IP 地址的作用就是帮助 IP 路由器实现转发和路由操作（确认网络流量的去向），从这个角度上讲，IP 地址和每个人的住址就十分类似了

正因如此，当设备链接到全球互联网中的时候，它们所使用的 IP 地址必须被协调直到不与网络上其它设备的 IP 地址重复：

  • IP 地址组会被 ISP 分配（allocated）给终端用户和组织；
  • 接受分配者又需要将 IP 地址指定（assign）给不同的设备；

基于以上，本小结将以如下方式组织：

  • 基本概念：

    • 接口（interface）是划分主机或路由器和物理链路的边界，每个接口都有自己独特的 IP 地址。因此 IP 地址在技术上的映射对象是接口，而不是路由器或者主机10；

      • 那些拥有多个 IP 地址，属于多个网络的主机就被称为多宿主主机（multihomed host）；
      • 由于路由器的作用就是将不同网络连接在一起，它必然属于多个网络也必然拥有多个 IP 地址；
      • 同一个子网内部所有接口的网络号都是相等的；

    • IPv4 地址的表示方式：

      • 长度为 32-bits（4 bytes），40 亿种可能性；
      • 点分十进制（dotted-decimal notation）：IP 地址中的每个 byte 用十进制形式表示（）以  形式分割；
      • 不可被主机随意选择：IP 地址的网络部分由其所连接的子网所决定；

  • IPv4 数据报格式：

    • 版本号；
    • 数据报总长度、头部长度；
    • 上层协议类型；
    • 服务类型；
    • TTL；
    • 目的地址、源地址；
    • 分片用字段；
    • 校验和；
    • 可选字段；

  • IP 地址的两种编址方式：

    • 分类编址（Classful Addressing）：IP 协议提出时所采用的编址方式；

      • 子网划分：根据 IP 地址前缀是否相同及相同的位数，将它们划分为不同的类别；
      • 子网掩码：网络会使用此字段判定诸如：节点是否属于指定子网之类的问题；
      • 变长子网掩码：上文提到的子网划分方式（固定子网掩码）可类比做将一块大蛋糕均匀地分成数块，也就是全部给切了。现在的变长子网掩码仅将部分 IP 地址根据需要划分出来作为一个子网，剩下很多没有被划分到任意一个子网中的 IP 地址；

    • 无类别域间路由（Classless Inter-Domain Routing，CIDR）：本质上是一种规范化，全球可见的的子网划分方式；

      • 地址聚合（Aggregation）：将地址连续的子网络，以路由器接口为核心组织在一起，从而减轻路由表负担；
      • 如何进行子网划分；

数据报格式

IPv4 的数据报由很多各部分组成：

  • 版本号（Version number）：

    • 长度：4-bits；
    • 作用：声明数据报的 IP 协议版本，帮助路由器解析 IP 数据报的剩余部分11；

  • 头部长度（Header length）：

    • 长度：4-bits；
    • 作用：考虑到 IPv4 的数据报的头部，可能会包含一些可选的数据，因此需要这一字段用以标识净荷（Payload）的起始位置12；

  • 服务类型（Type of service）：

    • 长度：8-btis；
    • 作用：用以区分不同的数据报。服务等级与数据报类型的划定是一个政策性的问题（并非技术性的，如 IPv4 和 IPv6 数据报的类型），需要由网络管理员来对路由器进行配置13；

  • 数据报长度（Datagram length）：

    • 长度：16-bits14；
    • 作用：用以标识数据报的总长度（头部加上净荷），单位是 ；

  • 校验、标记、分片偏移量（Identifier, flags, fragmentation offset）15：

    • 长度：32-bits；

    • 组成结构（计算的时候务必减去头部才能得到真正的载荷量）：

      • length：用以标识整个数据报的长度；
      • ID：用以标识小数据报所属的群体；
      • flagflag：用以标识此数据报之后还有没有其他的数据报（ 表示后方尚有数据报）；
      • offset： 用以标识本数据报中的净荷在大净荷中的偏移量（单位是  因此 ）

    • 作用：这部分数据的作用是用来对数据报进行IP 分片（IP fragmentation）操作的：

      • 发送方将一个大的数据报分割成数个、可以独立传输的小数据报，接收方再将他们组装起来；
      • 本质原因是不同的网络链路有着不同的最大传输单元（MTU），因此可能在某条链路上能进行传输的数据报到下一条链路上就需要被拆分；
      • 值得注意的是，经拆分的数据报仅会在目标主机上被重新组装，相关数据就储存在此处；

  • TTL（Time-to-live）：

    • 长度：8-bits；
    • 作用：为防止某个数据报一直在网络上被传输，这个字段中的值每被一个路由器进行处理就会递减 1。如果 TTL 递减为 0，路由器必须丢弃这个数据报；

  • 协议（Protocol）：

    • 长度：8-bits；
    • 作用：标识净荷应当被呈递给哪个传输层协议16；

  • 头部校验和（Header checksum）：路由器会将数据报的头部以  为单位进行拆分，随后使用 1s complement 计算这些数据的和，并将结果保存到此字段中。值得注意的是，考虑到数据报中的 TTL 每经过一个路由器就会改变（可选字段也可能会发生变化），因此每经过一个路由器，就需要重新计算数据报的校验和17：

    • 长度：16-btis；
    • 作用：路由器会对每一个接受到的数据报执行错误检查，如果计算出来的校验和与字段中所保存的值不符，路由器在一般情况下就会丢弃这个数据报；

  • 源地址和目标地址（Source and destination IP address）：

    • 长度：分别为 32-bit，源地址在前，目标地址在后；
    • 作用：供路由器使用；

  • 可选字段（Options）：这是一个已经被 IPv6 弃用的字段，主要是由于以下两个因素：

    • 由于可选字段的存在，无法预先确认净荷的开始位置；
    • 各个路由器处理数据报所需要用的时间有可能不同；

分类编址

分类编址（Classfun addressing）作为最先被广泛采用地址划分方式，将一段单播地址分成了两个部分18：

  • 网络部分：用以标识使用此单播地址的接口位于哪个网络中；
  • 主机部分：用以在特定网络中标识特定的接口（主机）；

考虑到不同子网络可能拥有不同数量的主机（各自拥有独立的 IP 地址），IP 地址的地址空间（根据字节的位数）被分成了五类：

  • A（0 开头）、B（10 开头）、C（110 开头）用于单播；
  • D（1110 开头）用于多播；
  • E（1111 开头）保留使用；

总的来说，A 类的网络部分占有一个字节、B 类的网络部分占有两个字节、C 类的网络部分占有三个字节：

不过，上表实际上是不准确的，任何一个子网中必须至少要减去两个地址：

  • 首个地址（主机号全 0）会被用作子网本身的地址；
  • 最后一个地址（主机号全 1），会被用作广播地址；
  • 0.0.0.0 会被用作 DHCP 客户端的初始地址；
  • 255.255.255.255 也会被用作广播地址；

下面是两个补充概念

  • 子网广播地址（subnet broadcast address，directed broadcast）是一种外部网络接口用于向指定子网内部所有设备广播数据报的地址，由于安全问题，很多路由器默认关闭对直连广播的转发操作19；

    • 这就是上边提到的 “最后一个地址”，主机号全 1 的地址；

  • 本地网络广播（local net broadcast）是一类特种网络，目的地址格式统一为 255.255.255.255。这种网络和直连广播类似，都是用于向子网内部广播数据报，不同之处在于这种数据报永远不会被路由器转发，因此只能在子网内部流通；

最后，IPv6 中没有任何广播地址，如果想要行使广播功能的话，IPv6 会使用多播地址

下面介绍的所有内容都是基于分类编址实现的，切莫与后边的 CIDR 搞混淆了

子网编址

子网编址（subnet addressing）允许某个地点将分配给它的地址空间（Class A, B, C）进一步的划分成两个部分：子网（subnet）号和主机号（host number）

本质上，子网编址仅仅只是在原来的 IP 地址中加了一个字段，没有改变其总长度：

子网编址具有以下三个性质

  1. 子网号对外不可见：子网号仅在网络点内部通用，其他网络点一般情况下不能知道本网络点的子网划分方式；
  2. 子网号也有保留地址：子网号的第一个和最后一个子网号不能被使用，因此对于子网号长度为  的子网来说，其最多只能容纳  个子网；
  3. 子网内设备前缀相同：同一个子网内部的所有接口子网号也相同；

注意，分裂编址本身的保留地址和子网的保留地址是不冲突的。也就是说对于 B 地址来说 xxx.xxx.0.0 来说，xxx.xxx.0.0 和 xxx.xxx.255.255 都会被保留。如果在这个 B 类地址空间中进一步划分出子网，如 xxx.yyy.0.0，那么 xxx.xxx.yyy.0 和 xxx.xxx.yyy.255 也会被保留。同时，地址空间 xxx.xxx.0.0 ~ xxx.xxx.0.255 中任何地址也是不能用的（广播地址同理）

子网掩码

子网划分

子网划分归结起来其实就这么几步：

  • 确认所需子网数目、各子网主机数目（舍弃首位子网号和舍弃首位主机号的过程相互独立，不要搞混了）：

    • 本质上来说，这是确定我们要在 ISP 给的网络号中，借出多少位的问题；
    • 考虑到子网号的首位两位不可用，因此如果划分出  位作为子网号，只能得到  个子网；
    • 考虑到主机号的首位两位不可用，因此如果划分出  位作为主机号，每个子网只能有  个主机；
    • 综合来看，无论是主机号还是子网号，至少需要划分两位出来（“Not a Legal Subnet Value”）；

  • 确认子网掩码：

    • 子网掩码实际上是根据上一步中所借的比特位数确定的，本质就是将二进制形式子网掩码表示成十进制；

  • 确认“幻数”：

    • 所谓幻数实际上是用以计算子网号的辅助工具，其计算方式是使用 256 减去子网掩码中最后一个非零字节对应的十进制数（更准确的讲，是将最后一个字节的网络号部分全部填上 1 之后得到的十进制数），上例的幻数就是 ；

无类别域间路由

分类编址被推广成了一种类似于 VLSM 的模式， 网络路由系统则被拓展以支持无类别域间路由（Class-less Inter-Domain Routing，CIDR）也称为构成超网（supernetting）：

  1. 以一种简便的方式分配规模从 255 到 65536 不等的子网络（介乎 Class B 到 Class C 之间），使 IP 地址块的大小不再受限；
  2. CIDR 掩码（CIDR mask）允许我们像类似于 VLSM 一样定义自己的子网号，不同之处在于 CLDR 掩码为网络上所有的路由系统可见（为此网路上所有的路由系统都需要能够识别 CIDR 掩码）；
  3. 前缀（Prefix）的出现完全代替了原来的分类编址和子网划分体系，两者现在合二为一，共同成为全局可见的、既适用于 IPv4 也适用于 IPv6 的网络前缀；

另外，有时候会看到诸如 10/10 的地址，实际上是将原地址的低位 0 比特，完整的写法是 10.0.0.0/10

IP 转发

IP 协议会从两个来源接收数据报：

  • 网络协议栈上层协议（传输层）；
  • 网络接口接受到的数据报（链路层）；

当网络层从网络接口中接受到数据报之后，它首先会检查两个事情：

  • 该数据报的目的 IP 地址是否是本机 IP 地址；
  • 该数据报的目的 IP 地址是否是某些特殊的地址（多播地址、广播地址）；

上述两个条件只要满足任意一条，主机就会参照数据报中的 Protocol（IPv4）字段或者 Next Header（IPv6）字段决定应当将数据报呈递给传输层的哪一个协议

如果上述条件都不满足：

  • 如果本机是路由器（或者说被设置为路由器），数据报会被转发；
  • 若非路由器，静默丢弃数据报（也有可能向源主机响应一条 ICMP 消息）；

而如果网络层是从上层协议中接受到数据报，那么：

  • 如果目标 IP 地址直接连接到主机（PPP）或者说目标主机与自己位于某一个共享网络中，主机可以直接将 IP 数据报发送给该主机（并不需要路由器）；
  • 此外的所有情况主机都会将数据报发送给一个默认路由器，令其代由自己分发数据报；

分组转发算法

分组转发算法实际上指的是根据数据报中的 ，推断应当将数据报转发到何处。从数据报的首部提取出目的地地址 ，将其与转发表中各项的  进行最长前缀匹配21：

  • 首先，路由器会检查目的地址是否直接连接到本机所在网络中，如果在，就直接交付（Direct Delivery）；

    • 如果匹配到条目的 Next-hop 是本机某个接口的地址，就代表是直接交付；
    • 否则代表目的网络地址并不位于直连网络中，执行间接交付（Indirect Delivery）；

  • 随后，匹配转发表中特定主机路由，若匹配到就交付到 Next-hop 所指示的下一跳路由器；

  • 随后，匹配转发表中的网络路由，若匹配到就交付到 Next-hop 所指示的下一跳路由器；

  • 最后，匹配转发表中的默认路由，如果连默认路由都匹配失败就会报错；

    • 默认路由的目的地址表项和子网掩码一般是 0.0.0.0；
    • 但是也有很多情况默认路由被严格限定；

ISP 的上层还有一个全球性的组织，互联网名称与数字地址分配机构（Internet Corporation for Assigned Names and Numbers，ICANN），其作用不仅包括为 IP 分配 IP 地址群，还包括：

  • 分配域名，解决域名冲突；

DHCP

当网络管理员获得 ISP 分配的 IP 地址块时，即可以选择手动为路由器配置 IP 地址，也可以选择使用动态主机配置协议（Dynamic Host Configuration Protocol，DHCP）令主机自动获取 IP 地址22

网管可以配置 DHCP，令其执行两种不同的地址获取方式：

  • 同一设备每次连接网络的时候都会获得一个相同的 IP 地址；
  • 同一设备在不同时候链接网络的时候会获得不同的临时 IP 地址（Temporary IP Address）；

除了 IP 地址，根据配置的不同，DHCP 还允许主机获得一些额外的信息：

  • 子网掩码；
  • 第一跳路由器的地址；
  • 本地 DNS 服务器的地址；
DHCP 服务器

DHCP 自动配置 IP 地址的能力减免了网管的许多繁重的工作，尤其是对于那些诸如学校、咖啡馆等用户来来往往的场景。如果网管需要手动为每个到来的用户配置 IP 地址（很显然在每个地方，任何一台需要接入网络的设备都需要一个 IP 地址），这将是一件极为繁重的工作

当然，如果一个子网络想要使用 DHCP 来减轻网管的负担，就必须在子网中接入一个 DHCP 服务器（或者说经由路由器将 DHCP 路由到这个子网中）：

DHCP 执行过程

新接入网络的主机（DHCP 客户端）如果想要经由 DHCP 协议获取 IP 地址，需要执行以下四步：

  • 寻找 DHCP 服务器：新到达某个子网络中的主机（DHCP 客户端）所需要做的第一件事就是找到这个网络中的 DHCP 服务器并与之通信，过程大致可以描述如下：

    • DHCP 客户端将DHCP 发现报文（DHCP discover message）放入 UDP 包中并通过端口 67 发送；
    • 网络层会将该 UDP 包封装为数据报，其目的地址是 255.255.255.255（即留作本地网络广播之用的地址），其源地址是 0.0.0.0；
    • 链路层会将这个数据报进一步封装为数据帧，地址自然是 FF-FF-FF-FF-FF-FF 这个 LAN 广播地址；

  • DHCP 服务提供：DHCP 服务器接受到发现报文之后，就会将DHCP 提供报文（DHCP offer message）广播到网络中（目的地址同样 255.255.255.255），考虑 DHCP 提供报文的内含物23：

    • 发现报文中的事务 ID（transaction id，xid）：用于匹配请求和回应；
    • 服务器分发的 IP 地址（Your IP address，yiaddr）；
    • IP 地址租约时间（IP Address Lease Time）这是 IP 地址的合法时间；

  • DHCP 请求：DHCP 客户端会从接受到的 DHCP 提供包中进行选择，并广播含有DHCP 请求报文（DHCP request message）的数据包；

  • DHCP 确认：服务器会发送含有 DHCP 确认报文（DHCP ACK message）的数据报，以确认请求参数；

注意，以上四个报文都是广播报文

地址转换（NAT）

专用网络（Private network）指的是使用专用地址空间，为局域网网络内部的设备分配地址的网络。同一个网络内的设备可以使用专用地址直接进行通信：

  • A 类专用地址：10.0.0.0 - 10.255.255.255，单个 A 类网络；
  • B 类专用地址：172.16.0.0 - 172.31.255.255，16 个连续 B 类网络；
  • C 类专用地址：192.168.0.0 - 192.168.255.255，256 个连续 C 类网络；

但是如果位于局域网内部设备想要和外网进行通信，就必须使用网络地址转换（Network address translation，NAT）将源地址为专用地址的数据报加以转换，之后才能将其发送到外界：

基于以上，不难得出专用地址的一个重要特性：位于不同子网络的设备，所使用的专用地址（IP 地址）可以重复24，这一特性在很大程度上减少了对 IP 地址数量的需求

NAT 路由器

NAT 路由器的作用是充当外网设备与内网设备通信的媒介

NAT 路由器在外网看来，与其说是一个路由器，倒不如说是一个拥有单一 IP 地址的设备：

  • 所有经过 NAT 路由器转发的数据报的源 IP 地址都是该路由器的 WAN 侧 IP 地址；
  • 所有发向该路由器的数据报的目标 IP 地址也都是该路由器的 WAN 侧 IP 地址25；

NAT 转换表

NAT 转换表能帮助路由器确定应当将数据报发送给内网中的哪个具体设备

NAT 转换表（NAT translation table）帮助 NAT 路由器判断应当将外网发送进来的数据包发送给内网的哪一个主机，其工作方式大致可以表述如下：

  1. 主机照常将数据报发送到局域网中：内网地址为 10.0.0.1 的主机将目标地址（128.119.40.186）和随机选定的端口号（5001）封装到数据报并发送到局域网中；

  2. NAT 路由器对数据报进行处理：路由器接受到数据报以后，会将其原来的端口号（5001）和源地址（10.0.0.1）替换成新的端口号（3345）和路由器 WAN 侧 IP 地址（128.119.40.186）：

    1. “新” 端口号实际上是路由器从 NAT 表中选的一个未被使用过的端口号；
    2. NAT 表中的条目由五个部分组成：外网 IP 地址、“新” 端口号、WAN 侧 IP 地址、源端口号、内网 IP 地址26；

  3. 外网服务器使用经处理的数据报中的信息响应：外网返回的数据报中的目标 IP 地址应当是路由器所填入的 WAN 侧 IP 地址（128.119.40.186），目标端口也是新端口（3345）；

  4. 路由器再次对数据报进行处理：使用该数据报的源 IP 地址（服务器的 IP 地址，128.119.40.186）在 NAT 转换表中的 “外网 IP 地址项” 中进行搜寻，将目标 IP 地址替换为内网 IP 地址（主机 IP 地址），端口号亦然。随后将该数据报转发到 LAN 中；

优势

使用 NAT 作为 LAN 和外界通信的接口存在着诸多优势：

  • 缩减所需 IP 地址数目，所有设备共同使用 WAN 侧 IP 地址作为对外 IP 地址；
  • 无需通知外界，即可修改内部设备的 IP 地址；
  • 更换 ISP 的时候无需修改网络内各设备的 IP 地址；
  • 网络内的设备不可显式寻址，提高了网络安全性；

IPv6

相比于 IPv4，IPV6 在以下三个方面做出了修改：

  1. 拓展地址空间：IPv6 的地址空间现在变为了 128-bit，此外 IPv6 中还引入了任播地址（anycast address）允许向指定组别中的主机定向投递数据报；
  2. 更高效的头字节：现在的 IPv6 头部被固定为 40-byte，删去了 Optional，使路由器能更高效地处理数据报；
  3. 流标签（Flow labeling）：使用流标签允许发送方对接收方要求更加实时或者更高质量的服务，比如说视频相关的数据报就能被视为流，e-mail 这种非实时的数据报就可以不被视为流；

帧格式

以下是 IPv6 数据报的格式：

  • 版本号（Version）：4-bit，显然其中的值应当是 6；

  • 流量类型（Traffic type）：8-bit，基本上和 IPv4 中的 TOS 字段（Type of Service）类似：

    • 可以标识此数据报在流中的优先级；
    • 可以标识此数据报相较于其他应用的数据报应当被优先处理；

  • 流字段（flow field）：20-bit，用以标识数据报的流；

  • 净荷长度（payload length）：16-bit，用以标识净荷的长度；

  • 下一头字节（Next header）：8-bit，用以标识净荷所属的协议；

  • 跳数限制（Hop limit）：8-bit，用以标识此包还能跳多少次；

  • 源地址和目的地址：各 128-bit，注意是 IPv6 地址；

改进之处

IPv6 从数据帧中剔除了下面三个字段：

  1. 分片/重组字段：IPv6 不允许中间路由器拆分数据帧；

    1. 如果数据报过大，路由器直接丢弃数据报并向发送方发送 “Packet Too Big” 这条 ICMP 错误信息；
    2. 实际上，将分片与组装这种耗时较长的工作完全交由主机处理能极大提升网络中 IP 转发的效率；

  2. 头部校验和：考虑到无论是链路层还是传输层都会校验数据，因此在网络层中对数据在此校验就会显得有些冗余（更多的是处于对网络效率的考量，毕竟在 IPv4 中每一跳都需要路由器计算一次校验和）；

  3. Options：IPv6 中将原来 Option 部分的信息单独抽取出来作为拓展头部（Extension header），这些多出来的信息会和其他高层协议的头部级联起来，跟随在 IPv6 头部的后边（IPv6 必须是第一个）：
