Title: 第二章前三节笔记 : 硬件支持

为使操作系统能在隔离不同用户和进程的同时，让用户进程能直接在硬件上运行，硬件至少需要支持以下三样东西：

  • 特权指令：处理器位于用户模式时，由于这些指令具有潜在不安全性，因此不能被执行。反之，如果处理器位于内核模式则可以安全执行这些指令；

  • 内存保护：处理器位于用户模式时，进程的任何试图对未经授权的内存区域的访问都将被禁止。反之，如果处理器位于内核模式则可以访问其它所有进程的内存区域；

  • 计时器中断：无论如何，内核都必须确保处理器控制权周期性地从应用程序手上被交还给内核；

    特权指令

仅能在内核模式但不能在用户模式中执行的指令被称为特权指令（Privileged instructions），它们是内核为了能完成其工作，所必须要执行的指令。例如：修改特权等级、调节内存访问、禁止/许可中断。很显然，这些指令不能被应用程序所执行

应用程序执行未经许可的行为（例如执行特权指令）将会引发处理器异常（Processor exception），不同于高级程序语言中的异常，处理器异常往往会使处理器直接将控制权交给（位于内核中的）异常处理器，它往往会直接中断引发此类异常的程序

内存保护

内存保护简单来说就是确保进程仅能读写自己的内存，不能读写操作系统和其它进程的内存

内存保护的意义

为什么内存保护是有意义的：

  • 内存保护能提升系统的可靠性：如果没有内存保护，一个程序的崩溃将很容易影响到另一程序；

  • 内存保护是抽象的一部分：在没有内存保护的时代，程序往往直接和内核进行交互，因此可以直接修改其中的数据。这将导致难以对内核代码进行更改，因为一旦更改内核，之前那些直接和底层数据结构进行交互的应用程序立刻就变得不能用了；
Base & Bound：实现

Base & bound 内存保护指的是每当处理器处理一条指令的时候，都会检查这条指令本身及其将要操作的数据在内存中的地址是否在 base 寄存器和 bound 寄存所划定的范围内。一旦超过这个范围，处理器会立即发出异常、暂停程序、将控制权转交操作系统内核。相关的技术细节如下：

  • Base & bound 依托于两个位于处理器内部的寄存器进行实现，操作系统保存进程时也会将这两个寄存器一并保存。简单来说，它们也属于状态寄存器的一种；

  • Base & bound 所划定的范围是物理地址空间中的一片区域，并不是虚拟内存空间中的一片区域；

值得注意的是，对于内核进程来说，不存在 Base & Bound 这两个概念（内核能访问系统中的所有内存）

Base & Bound：弊端

直接使用物理地址实行内存保护的 Base & Bound 存在非常多的问题（扩张、转换、共享、分片）：

  • 堆和栈的大小变动：由于 Base & Bound 为程序所分配内存的大小在程序被加载到内存的时候已经固定了，考虑到大小可能发生变动的堆和栈，究竟要使用 Base & Bound 为其划定多大的内存是一个很难确定的问题；

  • 内存共享：由于 Base & Bound 禁止不同进程访问对方的内存区域，诸如共享代码之类的事件可能很难实现；

  • 物理内存地址：当程序被编译和链接时，其全局变量和函数的地址都是相对于镜像文件（ELF）的开始部分相对决定的，因此当程序被加载到内存时，内核必须修改这些地址以确保程序的正常运行；

  • 内存分片：由于程序使用的是物理地址，因此每个程序在内存中所占据的区域必然是一块连续的区域。根据程序规模的大小不同，进程所占据的空间大小自然也不等。因此当计算机运行一段时间之后，可能会出现没有足够的连续内存供新进程被加载到内存中；

内存映射设备

内存映射设备（Memory-mapped device）指操作系统会通过读写特定位置内存与设备进行交互。具体来说，每个设备都会监视内存总线，一旦在其中发现分配给自己的地址就会启动相应的 I/O 操作

内存保护能为这种设备带来一定的好处，比如防止应用程序直接访问 I/O 设备

计时器中断

尽管在通常情况下应用程序的执行过程都不需要内核介入，但是发生异常情况时（死循环、越界访问）就需要内核执行一定的指令，判别是否应当停止应用程序的执行。当然，这其中最重要的是，如何确保在任何必要的时候，操作系统都能拿到处理器的控制权呢控制权？

硬件计时器（Hardware Timer）是一种实现方式。经过特定的一段时间之后（可能是以时间为单位，也可能是以指令的执行数目为单位）处理器就会中断并将控制权交给操作系统。有些技术细节需要注意：

  • 每个计时器仅会中断计算机中的一个核心，因此对于多核处理器来说，每个核心往往拥有独立的计时器；

  • 中断的周期往往以毫秒为单位，人类无法察觉；

  • 计时器只能被特权指令设置，应用程序无法更改其中的值；

另外，操作系统实际上是不知道程序是否正处在死循环中的，它唯一需要做的是确保能在死循环程序运行时询问用户：是否需要结束程序？

OS 的其它部分

值得注意的是，操作系统不仅仅包括内核，还包括诸如链接库和其它各种各样的程序。至于为什么不将这些部分也纳入内核，原因有二：

  • 第一，易化操作系统的调试。调试内核需要使用比内核更靠近硬件的调试器才能实现；

  • 第二，提高可依赖性。毕竟所谓受信任的代码越少就意味着错误越少，操作系统的可依赖性自然也就更强；
