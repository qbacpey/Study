Title: 第二章前三节笔记

内核抽象

保护（Protection）指的是：将具有潜在威胁性的用户和应用程序孤立出来，防止它们干扰其它应用程序和操作系统的正常运行。这项职责可被概括为四个方面：

  • 可靠性（Reliability）：无论应用程序做了什么，操作系统都要确保错误不会影响到其它程序或操作系统；

  • 安全性（Security）：考虑到应用程序具有潜在的威胁性，操作系统必须严格限制应用程序的行为；

  • 隐私性（Privacy）：操作系统必须严格限制应用程序和用户所能读取到的数据范围，防止它们在未经许可的情况下查看其它用户或程序的数据；

  • 公平的资源分配（Fair resource allocation）：公平有效的资源分配在某种程度上是和保护暗合的。简单来说，如果系统中的某一用户或者某一恶意应用程序占据了所有硬件资源，将直接导致其它程序乃至操作系统完全不能运行，因而对上述行为的禁止也应被归入 “保护” 的范畴；

操作系统内核（Operating system kernel）的工作便是实现 “保护”，因此内核必须成为计算机上最靠近底层的软件。作为计算机中唯一受信任（trusted）的软件，内核拥有对硬件的最高支配权，其它所有软件都不能像内核一样使用硬件的所有功能：

与操作系统内核相对的进程（Process）可以从两个方面进行定义：

  • 拥有严格权限限制的应用程序执行过程；

  • 对“受保护的执行过程”的一层抽象（保护由内核提供）；

言下之意就是：进程是应用程序在受内核保护、并且拥有严格限制情况下的执行过程更重要的是，进程的任何过程都需要操作系统内核的许可才能进行进程抽象

什么是进程？它与程序有何不同之处？

编译器会将高级语言源代码转变成机器码序列，并将其保存到被称为可执行映像（Executable image）的文件中可执行映像，该文件实际上就能被认为是程序（Program）

为了执行上述程序，OS 自然需要将程序从可执行映像中加载到物理内存中加载，使之成为上述程序的实例：进程（Process）。进程中除了有静态数据和代码（来自程序的可执行文件），还需要有分别用以保存局部变量和动态分配的数据的执行栈（Execution stack）和堆（heap）

当 OS 想要正式运行程序时，会设置处理器中的栈指针并（通过设置 PC 寄存器的方式）跳转到内存中程序第一行指令所在的位置

进程控制块

进程控制块（Process control block，PCB）是操作系统用于跟踪计算机进程状态所使用的数据结构，其中保存操作系统需要了解到的，关于某个进程的所有信息：

  • 进程在内存中保存的位置；

  • 可执行映像在硬盘中的位置；

  • 令程序开始执行的用户信息；

线程

对于那些拥有多个同时进行的任务的程序来说（这些任务亦称线程），每个任务都有着独立的的 PC 和栈，只不过与同一进程内的其它线程共享代码和数据

对偶模式

什么硬件允许操作系统有效地实现硬件抽象？

对偶模式操作（Dual-mode Operation）被提出的根本原因，是因为计算机需要处理两种不同类型的指令：

  • 来自用户的、可能存在非法行为的指令；

  • 来自内核的、必然合法的指令；

很显然对于前一种指令来说，内核需要对每一条指令执行权限检查，以确认其将要执行的操作是否合法；对于后一种指令来说，由于它们是可信赖的，因此无需检查，可以直接执行

如果对上述两种指令都执行检查，计算机的性能可能要打折扣。但如果都不检查而放任它们在硬件上直接执行，操作系统内核又难以履行“保护”这一职责，对偶模式的出现解决了这个问题

对偶模式操作由两种不同类型的处理器模式所构成：

  • 处于用户模式（User-mode）中的处理器在执行每一条指令之间，都会验证其对应的的操作是否被授权（检查进程是否被授权），即开启保护检查；

     

    处于内核模式（Kernel-mode）中的操作系统并不会执行保护检查；

    注意，这两种模式实际上是针对硬件定义的模式，实践中会使用处理器的状态寄存器（State register）来定义当前所处的模式状态寄存器

硬件支持

为使操作系统能在隔离不同用户和进程的同时让用户进程能直接在硬件上运行，硬件至少需要支持以下三样东西：

  • 特权指令：处理器位于用户模式时，由于这些指令具有潜在不安全性，因此不能被执行。反之，如果处理器位于内核模式则可以安全执行这些指令；

  • 内存保护：处理器位于用户模式时，进程的任何试图对未经授权的内存区域的访问都将被禁止。反之，如果处理器位于内核模式则可以访问其它所有进程的内存区域；

  • 计时器中断：无论如何，内核都必须确保处理器控制权周期性地从应用程序手上被交还给内核；

特权指令

仅能在内核模式但不能在用户模式中执行的指令被称为特权指令（Privileged instructions），它们是内核为了能完成其工作，所必须要执行的指令。例如：修改特权等级、调节内存访问、禁止/许可中断。很显然，这些指令不能被应用程序所执行

应用程序执行未经许可的行为（例如执行特权指令）将会引发处理器异常（Processor exception），不同于高级程序语言中的异常，处理器异常往往会使处理器直接将控制权交给（位于内核中的）异常处理器，它往往会直接中断引发此类异常的程序

内存保护

内存保护简单来说就是确保进程仅能读写自己的内存，不能读写操作系统和其它进程的内存

内存保护的意义

为什么内存保护是有意义的：

  • 内存保护能提升系统的可靠性：如果没有内存保护，一个程序的崩溃将很容易影响到另一程序；

  • 内存保护是抽象的一部分：在没有内存保护的时代，程序往往直接和内核进行交互，因此可以直接修改其中的数据。这将导致难以对内核代码进行更改，因为一旦更改内核，之前那些直接和底层数据结构进行交互的应用程序立刻就变得不能用了；

Base & Bound：实现

Base & bound 内存保护指的是每当处理器处理一条指令的时候，都会检查这条指令本身及其将要操作的数据在内存中的地址是否在 base 寄存器和 bound 寄存所划定的范围内。一旦超过这个范围，处理器会立即发出异常、暂停程序、将控制权转交操作系统内核。相关的技术细节如下：

  • Base & bound 依托于两个位于处理器内部的寄存器进行实现，操作系统保存进程时也会将这两个寄存器一并保存。简单来说，它们也属于状态寄存器的一种；

  • Base & bound 所划定的范围是物理地址空间中的一片区域，并不是虚拟内存空间中的一片区域；

值得注意的是，对于内核进程来说，不存在 Base & Bound 这两个概念（内核能访问系统中的所有内存）

Base & Bound：弊端

直接使用物理地址实行内存保护的 Base & Bound 存在非常多的问题（扩张、转换、共享、分片）：

  • 堆和栈的大小变动：由于 Base & Bound 为程序所分配内存的大小在程序被加载到内存的时候已经固定了，考虑到大小可能发生变动的堆和栈，究竟要使用 Base & Bound 为其划定多大的内存是一个很难确定的问题；

  • 内存共享：由于 Base & Bound 禁止不同进程访问对方的内存区域，诸如共享代码之类的事件可能很难实现；

  • 物理内存地址：当程序被编译和链接时，其全局变量和函数的地址都是相对于镜像文件（ELF）的开始部分相对决定的，因此当程序被加载到内存时，内核必须修改这些地址以确保程序的正常运行；

  • 内存分片：由于程序使用的是物理地址，因此每个程序在内存中所占据的区域必然是一块连续的区域。根据程序规模的大小不同，进程所占据的空间大小自然也不等。因此当计算机运行一段时间之后，可能会出现没有足够的连续内存供新进程被加载到内存中；

内存映射设备

内存映射设备（Memory-mapped device）指操作系统会通过读写特定位置内存与设备进行交互。具体来说，每个设备都会监视内存总线，一旦在其中发现分配给自己的地址就会启动相应的 I/O 操作

内存保护能为这种设备带来一定的好处，比如防止应用程序直接访问 I/O 设备

计时器中断

尽管在通常情况下应用程序的执行过程都不需要内核介入，但是发生异常情况时（死循环、越界访问）就需要内核执行一定的指令，判别是否应当停止应用程序的执行。当然，这其中最重要的是，如何确保在任何必要的时候，操作系统都能拿到处理器的控制权呢控制权？

硬件计时器（Hardware Timer）是一种实现方式。经过特定的一段时间之后（可能是以时间为单位，也可能是以指令的执行数目为单位）处理器就会中断并将控制权交给操作系统。有些技术细节需要注意：

  • 每个计时器仅会中断计算机中的一个核心，因此对于多核处理器来说，每个核心往往拥有独立的计时器；

  • 中断的周期往往以毫秒为单位，人类无法察觉；

  • 计时器只能被特权指令设置，应用程序无法更改其中的值；

另外，操作系统实际上是不知道程序是否正处在死循环中的，它唯一需要做的是确保能在死循环程序运行时询问用户：是否需要结束程序？

OS 的其它部分

值得注意的是，操作系统不仅仅包括内核，还包括诸如链接库和其它各种各样的程序。至于为什么不将这些部分也纳入内核，原因有二：

  • 第一，易化操作系统的调试。调试内核需要使用比内核更靠近硬件的调试器才能实现；

  • 第二，提高可依赖性。毕竟所谓受信任的代码越少就意味着错误越少，操作系统的可依赖性自然也就更强；

模式转换

由于在用户模式和内核模式之间相互转换时常发生，因此提供安全且高效的模式转换组件就显得非常有必要了

从用户到内核

以下三种情况会导致用户模式转变为内核模式：

  • 中断（Interrupt）；

  • 处理器异常（Processor exception）；

  • 系统调用（System call）；

中断是异步（Asynchronously）发生的，它满足以下两个条件：

  • 为外部事件所引发；

  • 可能在任何用户模式指令执行之后就切换到内核模式；

同步发生的处理器异常和系统调用则与之相反，作为同步事件，它们会被处理器异常所驱动。俘获（trap）指的就是同步的、从用户模式转变为内核模式的过程

中断

中断（Interrupt）实际上就是一个用以通知处理器的异步信号，告诉它外部发生了某件需要其处理的事情。处理器执行指令的过程中会时刻注意是否存在中断信号。若存在，处理器将会按如下步骤处理中断：

  • 处理器选择完成当前指令亦或是暂停当前指令的执行（例如向流水线中发送 nop）；

  • 处理器保存当前执行状态（而不是取下一条指令）；

  • 处理器开始执行位于内核中的中断处理程序（Interrupt handler）；

中断的类型

中断的类型各式各样，相应的处理方式也各不相同：

  • 由计时器引发的中断：

    • （位于内核中的中断）处理器检查当前进程是否响应于用户的输入，以便断定其是否陷入死循环；

    • 有时处理器也需要实行进程切换，以确保资源分配的公平性；

    • 一切照常时，计时器异常处理器（timer handler）会在指令的中断处恢复执行，此时的中断对用户进程是透明的（用户进程无法感知中断是否发生）；

  • 与 I/O 设备相关的中断：

    • I/O 设备（以太网、WiFi、硬盘、交互设备）在接受到输入或请求完成时，都会向处理器发送中断；

此外，中断除了来源于计时器、I/O 设备，还有可能来自于同一台计算机上的其它处理器。内核会使用这些在处理器间传递的中断，让不同处理器协作（对于并行程序来说，这是一个很重要的特性）

轮询

处理器也可以采用相对主动的方式应对 I/O 中断：轮询（Polling）

此时内核会不断循环轮询检查每个 I/O 设备中是否有事件发生

I/O 中断

考虑到当今 I/O 设备的速度和处理器的速度相差甚远，因此提升计算机性能的关键就在于确保每时每刻 I/O 设备都在工作

过去一种不那么高效的 I/O 设备使用方式是让 I/O 设备在同一时间仅执行一项 I/O 操作。I/O 设备完成请求就会向处理器发送中断，中断处理程序处理完中断之后再给 I/O 设备下一项任务。很显然，在中断处理程序处理中断的空档，I/O 设备是处于空闲状态的

如果对性能有着更高的要求，操作系统可以选择使用一个环形请求队列让 I/O 设备接收并处理请求。队列中的每个条目被称为缓存描述符（Buffer descriptor），每个缓存描述符描述了一项 I/O 操作，其中包含所请求的 I/O 操作以及用来放置数据的缓存的位置DMA

处理器异常

处理器异常（Processor exception）是一种使处理器从用户模式转变为内核模式、由用户程序所引发的硬件事件：

  • 用户行为不当：多数情况下处理器异常都是由于用户行为不当而引起的，如内存越界、除 0、用不对齐的地址访问内存，此时中断处理程序往往会选择直接终止进程；

  • 调试程序：但处理器异常也能被用来执行程序调试，比如说由断点或者说程序中某条俘获指令导致的处理器异常，此时中断处理程序会将控制权转交给调试器；

处理器异常的作用远不止保护和调试，它还是虚拟化得以实现的基础，包括：虚拟化处理器架构、虚拟化整个底层硬件、还有虚拟化内存：

  • 虚拟化处理器架构：早期的处理器限于性能（和指令集），不能执行浮点数运算。处理器异常可以将性能为代价，在一定程度上解决这个问题。当处理器执行到浮点运算语句的时候，它会发出处理器异常，此时的内核不会像往常一样直接中断进程的执行，而是选择去执行一段浮点数运算程序，执行完毕立即恢复原程序的执行；

  • 虚拟化底层硬件：即虚拟机的一部分运行逻辑，当客户操作系统想要执行特权指令的时候（往常内核会直接中断进程的执行），接受到处理器异常的宿主内核会代其执行相应的特权指令并继续客户操作系统的执行；

  • 虚拟化内存：从实现上讲，这实际上是将处理器设置成“一旦读写到指定内存就引发处理器异常”。如是当程序使用经虚拟化拓展出的内存时，内核就会立即从硬盘中取出数据放到相应的位置，随后恢复程序的执行；

系统调用

系统调用（System call）在本质上仅仅是提供自内核的、一段能被用户调用的程序，它允许用户进程主动地将控制权转交给内核，让其代用户执行某些操作

大多数的处理器都是通过一些特别的语句实现系统调用的，如 trap 和 syscall。但也有一些特例，某些系统中会直接执行一些拥有 “非法” Opcode 的指令来实现这一点

系统调用的过程和之前的两种模式转换方式十分类似，都是让内核在中断之后执行一段特别的程序。不过，为了防止用户程序被意外影响到，系统调用所在的位置必须在内核中——禁止用户程序直接访问其对应的内存位置

尽管如此，在用户眼中，系统调用和其它第三方库相差无几，无非就是按照接口声明调用函数，随后得到既定结果而已

从内核到用户

下面简单介绍几种简单的、从内核模式切换到用户模式的场景：

  • 启动新进程：为启动新进程，内核需要将程序拷贝到内存、将 PC 设置为进程第一条指令所在位置、将栈指针设置到用户栈的底部，随后才会切换到用户模式；

  • 从中断、处理器异常、系统调用中恢复；

  • 切换到不同的进程：有些时候内核需要切换进程的执行（比如说计时器中断），这其中 PCB 就充当了保存进程状态的重要作用；

  • 用户层上行调用（upcall）；

[可执行映像]  编译器同时也会将源代码中定义的静态数据及其初值在编译时一并保存到可执行映像中
[加载]  当然，无论是操作系统还是编译器，都得预先被加载到内存中才能完成这一切
[状态寄存器] 处理器中控制当前模式的标记（flag）不能被应用程序代码直接进行访问，与之相关的修改操作往往是某些指令的连带效果使然（并非直接导致）。譬如说当中断发生时，硬件必须自动地将状态寄存器中的值保存到内存中（中断处理程序很可能会直接修改状态寄存器中的值）；另外，状态寄存器中还有一些被称为条件码（Condition code）的标记，会被算术指令带来的副作用所修改，以便条件分支语句的编码更加紧凑
[控制权] 操作系统不止在异常情况下需要控制权，在一些跨进程的活动中也需要控制权
[轮询]  循环和忙碌等待都是轮询的实现方法
[DMA]  值得注意的是，I/O 设备使用 DMA 访问内存，而一项 I/O 操作很可能涉及到多项 DAM 操作
