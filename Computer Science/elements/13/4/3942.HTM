Suppose one file contains a global array: 

char my_arr[256];

And soothed by the seeming equivalence between arrays and pointers, the programmer that wants to use it in another file mistakingly declares as:

extern char* my_arr;

When he tries to access some element of the array using this pointer, he will most likely get a segmentation fault or a fatal exception (the nomenclature depends on the OS). Understanding why this happens is left as an exercise to the reader 

在汇编中，变量的名字实际上是对变量所在内存地址的指代，或者说变量的名字本身实际上就是变量的地址，因此要使用变量的名字去内存中寻找保存在该地址中的值，也就是所谓变量的值

那么如果像这样使用 extern 的话，会不会让运行时将 my_arr 当成一个变量呢？如果它被当成一个指针变量的话，那么如果运行时要访问指针所指向的内容，首先会解引用这个指针，但是由于这个指针实际上是数组，其内容并不是地址，而会是数组的内容，所以在这种情况下如果需要访问指针所指向的内容，实际上相当于解引用数组的元素而不是数组的地址

当然上面只是猜测，具体还是要明确 extern 的意义才行

Hint: look at the first assembly listing in this article. How will the element be accessed via the pointer? What's going to happen if it's not actually a pointer but an array?

看来确实是这样：

    char a = array_place[7];

    char b = ptr_place[7];

当运行时使用 extern 变量访问数组时，编译出来的代码会使用类似于上述汇编中第二种方式里边提到的那样，先获取变量的值（即所谓的 char 指针）随后递增并解引用（编译器会认为变量此时的内容应该是元素）但是由于这个“变量”实质上是一个地址标签，因此其中的值并不是数组的地址，而是货真价实的——数组内容，也就是说，我们将数组内容当成数组地址解引用了

Suppose one file contains a global array: 

char my_arr[256];
extern char* my_arr;

为什么使用下面的 my_arr 访问数组元素会出现 segmentation fault？

编译器使用数组名访问元素和指针访问元素所生成的汇编代码不相同。考虑到在汇编中数组的名字本身就是数组首元素的地址（指针的名字则是指针的所在地址，而指针中所保存的值才是变量的地址）因此尽管源代码中使用指针和数组名能混用语法，但是实际为这两者生成的汇编代码是不相同的
